# Avito Help Search

Для каждого запроса из `test.f` ранжирую статьи справки и возвращаю
10 `article_id`. Целевая метрика — `MAP@10`.

## Результаты

| Подход | OOF MAP@10 | TestSys |
|---|---:|---:|
| Popularity | 0.3159 | 0.27 |
| Stemmed BM25 | 0.3187 | — |
| Query kNN | 0.6008 | — |
| E5 + char kNN | 0.6035 | — |
| LightGBM LambdaMART | 0.6910 | — |
| E5 + LightGBM blend | 0.6928 | 0.603060 |
| Noise/spell XGBoost | 0.6855 | — |
| Noise/spell final blend | 0.6960 | 0.600539 |
| AutoML-selected XGBoost | 0.6824 | — |
| **Final power blend** | **0.6951** | **0.607908** |

## EDA и эксперименты

В данных 793 статьи, 500 calibration-запросов и 500 test-запросов. Медианный
запрос содержит 10 слов. У 221 запроса больше одной правильной статьи, максимум
— четыре; в calibration встречается 79 разных целевых статей. Поэтому одной
классификации недостаточно: сначала собираю кандидатов по текстовой близости,
затем ранжирую пары запрос–статья.

Проверил popularity, BM25, stemmed BM25, word/char TF-IDF, query kNN,
LinearSVC, E5 kNN, LambdaMART, XGBoost и несколько rank-blend. Лексические
модели полезны как независимые признаки, но отдельно дают низкий MAP@10.
Query kNN и E5 лучше переносят разметку между похожими формулировками.
Основной прирост дал LambdaMART. Noise/spell-бленд вырос на OOF до `0.6960`,
но получил `0.600539` на TestSys, поэтому в финале оставил его признаки, а не
самостоятельный прогноз. AutoML и cross-encoder добавили небольшой устойчивый
прирост поверх основной ветки.

## Подход

1. Из HTML извлекаю видимый текст, заголовки, ссылки, `alt` и начало статьи.
   Отдельно строю stemmed-представление.
2. Получаю признаки из BM25, word/char TF-IDF, query kNN, линейных моделей,
   профилей запросов и связей между статьями.
3. Для пар `запрос — статья` считаю score-, rank-, overlap- и length-признаки.
   Два LightGBM LambdaMART обучаю с разными seed.
4. Локальная `multilingual-e5-small` кодирует запросы. Метки переношу от
   20 ближайших calibration-запросов, смешивая E5 cosine и char TF-IDF как
   `0.85 / 0.15`. Степень 8 усиливает близких соседей, деление на частоту
   метки в степени `0.3` ослабляет перекос к самым частым статьям.
5. Убираю приветствия и вводные фразы, исправляю опечатки по словарю статей.
   Исходный запрос сохраняю, новые варианты добавляю как отдельные признаки.
6. FLAML перебирает LightGBM и XGBoost по `NDCG@10`. Найденную конфигурацию
   XGBoost фиксирую в коде, поэтому для финального запуска AutoML не нужен.
7. Cross-encoder пересортировывает только top-10 основной системы. Итоговый
   power blend использует веса `0.85 / 0.12 / 0.03` для основной системы,
   AutoML-XGBoost и cross-encoder; степени рангов — `0.5 / 1.5 / 1.0`.

## Валидация и ошибки

Использую 5-fold nested OOF с `shuffle=True` и `random_state=42`.
Supervised query-признаки для каждого fold строятся только по train-части.
Веса и степени финального бленда выбраны по nested OOF. Оставлена самая
простая комбинация, которая улучшила основную систему на каждом из пяти
фолдов: `0.6928 → 0.6951`. На TestSys результат вырос с `0.603060` до
`0.607908`.

Основные ошибки возникают на редких статьях, коротких неоднозначных запросах
и вопросах с несколькими намерениями. Char TF-IDF и исправление опечаток
помогают с шумным вводом, E5 — с разговорными перефразировками, а ranker
объединяет эти сигналы.

Использую только выданные Feather-файлы и локальные open-source модели
`intfloat/multilingual-e5-small` и `DiTy/cross-encoder-russian-msmarco` с
зафиксированными ревизиями. Внешние API, ручная разметка `test.f` и ответы
для отдельных `query_id` не используются.

## Запуск

Все команды и ноутбуки запускаются из корня репозитория. Проверенное
окружение — Python `3.14.6`; версии библиотек зафиксированы.

```bash
python -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt
python run.py
```

При первом запуске скачиваются зафиксированные веса E5 и cross-encoder; дальше
модели работают локально. `run.py` проверяет формат и сохраняет
`outputs/answer.csv`. Нейросетевые модели запускаются отдельно от LightGBM и
XGBoost, чтобы нативные библиотеки не конфликтовали в одном Python-процессе.
В ноутбуках `00–08` сохранение submission закомментировано, в
`09_final_pipeline.ipynb` — включено.
